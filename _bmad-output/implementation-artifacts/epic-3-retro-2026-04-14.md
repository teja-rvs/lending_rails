### Deep Story Themes (from story records)

**What worked consistently**

- **Vertical slicing and explicit seams:** Each story extended the same canonical application workspace (`loan_applications#show`) instead of parallel pages. Stories 3.1–3.5 repeatedly called out “do not pull Story X forward,” which kept scope honest.
- **Server-side truth:** Workflow order, active step, eligibility, and final decisions stayed in services and models; UI remained HTML-first with clear flash/render feedback.
- **Reuse of read models:** `Borrowers::HistoryQuery` was reused for borrower lending context on the application page (3.4); list filtering used a dedicated `LoanApplications::FilteredListQuery`.
- **Audit and integrity:** `paper_trail` on applications (and steps where added), borrower snapshotting at creation (3.1), and transactional create + workflow initialization (3.2) matched the architecture expectations.

**Review and quality patterns**

- Code review surfaced concrete fixes that were tracked to completion, including: validation gaps (e.g. non-positive requested amounts), **application number allocation under concurrency**, active-step display when the workflow was complete, factory data matching canonical workflow, **search dropping active status filter** on the application list, **blocking review-step mutations after a final application decision**, and **stale “later story” copy** on the show page after workflow completion.
- Automated coverage expanded with each slice (request, service, system), and story completion notes reference full-suite runs—supporting confidence in regression safety.

**Technical debt and follow-ups called out in stories**

- **Application number generation:** Review noted allocation is not concurrency-safe; acceptable for spike/MVP but worth a dedicated hardening pass if volume or parallel creates increase.
- **PRD/planning gap on fixed step list (FR17):** Addressed in implementation via a single canonical workflow definition (`ReviewStep` / fixed sequence), but product docs could stay aligned with that single source of truth.

### What Went Well

- Story order matched the natural dependency chain; each story built on the previous without rewriting foundations.
- Fixed MVP workflow (`history_check` → `phone_screening` → `verification`) was centralized, idempotent, and backfilled safely for pre-existing applications.
- Final decisions (3.5) respected strict approval rules (all steps approved, application `in progress`) while keeping reject/cancel available before finals, matching the documented intent.
- Epic 2’s emphasis on **thin controllers, query/service seams, and calm operational UX** carried through cleanly.

### Challenges

- Several fixes were **UX and correctness details** (filters, breadcrumbs, messaging) rather than structural failures—good signal that the architecture was stable and issues were localized.
- **Concurrency and edge cases** (APP numbers, completed workflow vs. active step) are classic “MVP then harden” items; they are documented for follow-up rather than ignored.

### Key Insights

1. **One workspace, many stories:** Investing in a single canonical application page paid off; later stories added sections (workflow, borrower context, decisions) instead of new navigational dead ends.
2. **Guardrails in the story text matter:** Explicit “out of scope” lists prevented loan creation, disbursement, and final decision actions from landing in the wrong story.
3. **Review feedback is part of delivery:** Treating review patches as first-class work kept quality and UX aligned with the wireframe/list patterns used elsewhere (e.g. borrowers list).

## Next Epic Preparation (Epic 4)

### Preview

**Epic 4: Loan Setup, Documentation, and Disbursement Control** — starts with **Story 4.1: Create a loan from an approved application.**

### Dependencies on Epic 3

- **Approved application** status and linkage to borrower are the natural handoff; Story 3.5 explicitly did **not** create loans—Epic 4 owns that transition.
- **Blocking semantics:** `approved` applications remain blocking for new applications via existing borrower eligibility rules until the loan lifecycle resolves—aligned with PRD-style operational control.
- **Traceability:** Final statuses, list filters, and history surfaces support finding approved/rejected/cancelled applications for operations and downstream work.

### Preparation tasks (recommended)

1. **Confirm Epic 4.1 story scope** with the same strict “no disbursement in 4.1” boundary already in epics.md.
2. **Hardening backlog (non-blocking for kickoff):** application number allocation strategy if parallel creates become realistic.
3. **Doc alignment:** Ensure planning artifacts name the same fixed review steps as code (single canonical list).

### Significant discoveries

- **No change required to Epic 4’s fundamental assumptions** based on Epic 3 delivery: approval remains distinct from loan creation; pre-disbursement work stays in Epic 4 stories as written.
