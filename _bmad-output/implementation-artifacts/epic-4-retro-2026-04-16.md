## Deep story themes (from story records)

### What worked consistently

- **Single loan workspace (`loans#show`) as the spine:** List and detail flows extended one surface across 4.2–4.5 (filters, forms, documentation, readiness, disbursement) instead of fragmenting into unrelated pages—consistent with Epic 3’s “one application workspace” lesson.
- **Domain services and AASM:** Mutations live under `Loans::*`, `Documents::*`, `Invoices::*`; lifecycle transitions use AASM; controllers stay thin (`find → service → redirect + flash`).
- **Explicit story boundaries:** Stories repeatedly called out non-goals (no `double_entry` until 4.5, no readiness until 4.4, etc.), which kept scope disciplined.
- **Test discipline:** Story files cite full `bundle exec rspec` runs with growing example counts (195 → 312), supporting regression safety across a large vertical slice.

### Review and quality patterns

Recurring review-driven fixes (examples):

- **Serialized allocation:** Loan numbers (4.1), invoice numbers (4.5)—addressing the same class of concurrency risk called out for application numbers in Epic 3.
- **Data migration safety:** Backfill borrower snapshot columns before validations (4.1).
- **UI/domain alignment:** `Loan#next_lifecycle_stage_label` for overdue (4.1); interest-mode switching clearing stale opposing values (4.2); post-disbursement readiness panel showing historical context vs “blocked” (4.4); post-disbursement disbursement summary vs checklist (4.4, 4.5).
- **Infrastructure catch:** Removing a broken `shadcn` Stimulus bootstrap so client-side toggles work in the browser (4.2).

### Technical debt and follow-ups surfaced in stories

- **Application-number concurrency (Epic 3):** Not replaced by loan/invoice serialization; remains a **borrower application** backlog item if parallel creates become realistic.
- **Disbursement system spec:** Story 4.5 noted end-to-end browser flow deferred (request/service coverage strong); optional **add system spec** when driver setup is a priority.
- **Operational prerequisites:** Story 4.4 explicitly avoided inventing bank-details/charges columns—if the product later requires them, plan a **PRD/schema** increment before treating them as readiness gates.

### Testing and quality insights

- **double_entry + RSpec:** Resolved `LockMustBeOutermostTransaction` by structuring `Loans::Disburse` with `DoubleEntry.lock_accounts` as the outer boundary—worth documenting as the pattern for future money-moving stories (Epic 5 payments).
- **Environment friction:** Debug logs note PostgreSQL/Docker availability during development—keep **DB up** as a documented prerequisite for CI and local runs.

## What went well

- End-to-end **pre-money vs post-money** boundaries stayed clear: edits and documents locked after disbursement; money events go through `Loans::Disburse`.
- **Readiness as a first-class domain concept** (`Loans::EvaluateDisbursementReadiness`) avoids duplicating validation rules and keeps UI and server aligned.
- **Document history** (active vs superseded) matches FR76-style integrity without hard deletes.

## Challenges

- **Integration complexity rose** in 4.5 (AASM + invoice + `double_entry` + locking); debugging nested transactions required careful layering—captured in dev notes for the next money-moving change.
- **MVP scope vs real-world fields** (e.g. bank details) remains an explicit gap to manage with stakeholders before it becomes a surprise.

## Key insights

1. **One vertical spine beats many mini-features:** The loan detail page carried documentation, readiness, and disbursement without losing orientation.
2. **Review patches are part of delivery:** Serialization, migration backfills, and UI state fixes were not “extra”—they protected production-shaped behavior.
3. **Money-moving stories need explicit transaction boundaries** when bookkeeping gems participate—apply the `Loans::Disburse` lesson to payment completion in Epic 5.
