## Deep story themes (from story records)

### What worked consistently

- **Action-first dashboard design:** Story 6.1 delivered a dashboard that leads with actionable counts (pending applications, overdue payments, active loans) rather than passive summaries — operators land on what needs attention.
- **Drill-through navigation pattern:** Story 6.2 extended the dashboard counts into filtered list views with explicit filter context banners, clear-filter CTAs, and helpful empty states per filter combination.
- **Cross-entity search:** Story 6.3 unified search across borrowers, loans, applications, and payments — operators can find any record from a single search bar with linked drill paths.
- **PaperTrail audit trail:** Story 6.4 delivered `has_paper_trail` across all operational models with `whodunnit` tracking, `ActivityTimelineComponent` for version display, and `DeletionProtection` concern preventing hard deletes.
- **Derived state integrity:** Story 6.5 implemented facts-not-toggles for overdue derivation, loan closure from completed payments, borrower snapshots on loans and applications, and immutability guards on completed/locked records.
- **Test coverage:** Each story maintained comprehensive RSpec coverage (model, service, request, component, and system specs) with growing example counts across the full suite.

### Review and quality patterns

Recurring review-driven fixes:

- **Serialized allocation consistency:** Loan and invoice numbers used `LOCK TABLE` for concurrency safety; application numbers did not — resolved during this retrospective.
- **Snapshot immutability:** Borrower name/phone snapshots on `LoanApplication` and `Loan` protected by `before_update` guards against post-creation mutation.
- **ViewComponent reuse:** `StatusBadgeComponent`, `ActivityTimelineComponent`, and `EmptyStateComponent` used consistently across all operational views.
- **Query object pattern:** `FilteredListQuery` classes for each entity keep controllers thin and read logic testable in isolation.

### Technical debt and follow-ups surfaced in stories

- **No pagination:** All list views load full result sets; acceptable at current scale but will need pagination for production volumes.
- **DeletionProtection bypasses:** The concern raises `ActiveRecord::ReadOnlyRecord` but can be bypassed with `delete` vs `destroy` — documented in deferred-work.
- **No phone number index:** `borrowers.phone_number` column lacks a database index; search performance will degrade at scale.
- **GET requests mutating DB:** `PaymentsController#index` calls `Payments::DeriveOverdueStates.call` on every GET — a side-effecting read that should move to a background job.

### Testing and quality insights

- **System specs proved valuable:** `full_lifecycle_spec.rb` and entity-specific system specs caught integration issues that unit/request specs missed.
- **SimpleCov thresholds:** Global coverage thresholds (80% line, 50% branch) report below target when running individual spec files; full suite passes — this is a reporting artifact, not a real gap.
- **Factory interdependencies:** Payment factories depend on loan factories which depend on borrower factories; deep factory chains slow test setup but accurately reflect domain constraints.

## Retrospective-discovered fix

### Out-of-order installment completion guard

**Problem:** `Payments::MarkCompleted` allowed any pending/overdue payment to be completed regardless of installment sequence. Installment #5 could be completed while #1–#4 remained outstanding.

**Fix:** Added `BLOCKED_OUT_OF_ORDER` guard inside the pessimistic lock block in `Payments::MarkCompleted`. The `earlier_installments_incomplete?` check queries `loan.payments` for any installment with a lower number that isn't completed. The check runs under `with_lock`, preventing concurrent race conditions.

**Files changed:**
- `app/services/payments/mark_completed.rb` — added constant, guard clause, and private query method
- `spec/services/payments/mark_completed_spec.rb` — 3 new test cases (blocks out-of-order, allows when earlier completed, allows first installment)
- `spec/requests/payments_spec.rb` — 1 new test case (HTTP 422 with flash message)

**All tests pass:** 17 service specs, 41 request specs.

## What went well

- **Vertical slice discipline held through 6 epics:** Dashboard, filters, search, audit, and integrity all extended existing patterns (workspace surfaces, thin controllers, domain services, AASM) rather than introducing new architectural layers.
- **PaperTrail + whodunnit adoption was smooth:** One concern, one component, consistent integration across all auditable models.
- **Derived states from facts:** Overdue derivation, loan closure, and snapshot immutability all follow the "recorded facts, not manual toggles" principle established in earlier epics.
- **Previous retro commitments fully cleared:** All three Epic 4 carryovers were resolved, leaving no accumulated technical debt from retrospective action items.
- **Test suite maturity:** The project has comprehensive coverage across model, service, request, component, and system spec layers.

## Challenges

- **Deferred work accumulation:** The `deferred-work.md` file lists multiple items (pagination, phone index, GET mutation, deletion bypass) that are individually low-risk but collectively represent production-readiness gaps.
- **Factory chain complexity:** Deep factory dependencies (payment → loan → borrower) make test setup verbose and slower; no immediate fix without compromising domain accuracy.
- **SimpleCov threshold reporting:** Per-file runs report below threshold due to partial coverage calculation — misleading but not a real quality issue.

## Key insights

1. **Sequential enforcement belongs in the service layer:** The out-of-order payment gap showed that AASM state transitions alone don't enforce business sequencing — explicit domain guards are needed for ordering invariants.
2. **Retrospective commitments must be tracked to closure:** The Epic 4 carryovers lingered until this final retrospective forced resolution. Earlier follow-through would have caught the application-number gap sooner.
3. **Deferred work needs a triage cadence:** Items in `deferred-work.md` should be reviewed per-epic, not just at project end, to avoid a growing tail of production-readiness issues.

## Action items

### Technical (recommended for production readiness)

1. **Add pagination to all list views** (borrowers, applications, loans, payments).  
   **Priority:** High for production  
   **Rationale:** Current full-result-set loading will degrade with realistic data volumes.

2. **Add database index on `borrowers.phone_number`.**  
   **Priority:** Medium  
   **Rationale:** Phone search is a primary lookup path; missing index will cause full table scans.

3. **Move `DeriveOverdueStates` out of GET requests** into a scheduled background job.  
   **Priority:** Medium  
   **Rationale:** GET requests should be side-effect-free; current approach mutates DB state on every payments index load.

4. **Audit `DeletionProtection` coverage** to ensure `delete` (bypassing callbacks) cannot be used accidentally.  
   **Priority:** Low  
   **Rationale:** Current `before_destroy` guard doesn't protect against `delete`; consider database-level constraints for critical records.

### Process

5. **Triage `deferred-work.md` before any production deployment.**  
   **Owner:** Bob (SM) / RVS  
   **Rationale:** Items are individually manageable but collectively represent gaps that should be consciously accepted or addressed.

### Team agreements

- The **facts-not-toggles** principle for derived states is now a project convention — any new lifecycle behavior should follow this pattern.
- **Server-side guards for business invariants** (sequencing, locking, concurrency) are mandatory; UI-only enforcement is insufficient.
