Low-fidelity page set

lending_rails wireframes

Full-page wireframes derived from the completed UX specification, the chosen Direction 1: Ops Console, and the MVP lending workflows. These are intentionally low-fidelity so layout, hierarchy, and page relationships stay easy to review.

Shell

Left navigation, top-page heading, metric-first overview, calm operations layout.

Priority

Repayment triage, clear lifecycle state, linked records, guarded money actions.

Coverage

Login, dashboard, borrower/application/loan/payment/invoice pages and key forms.

How to review

These wireframes are meant to answer three things: what information each page carries, what the main action is, and how the admin moves from overview to detail without losing context.

  • Check whether each page has one obvious primary action.
  • Check whether status, lineage, and next-step cues are visible early.
  • Check whether dashboard, list, and detail pages feel like one system.
  • Use this set as a base for higher-fidelity screen design or implementation.

1. Login

Simple entry with product name, email, password, and clear recovery path.

auth
Primary action

Log in

Outcome

Land directly on dashboard

Polished but restrained entry screen. No marketing clutter, just product identity, form, and calm error handling.

2. Dashboard

Action-first operations overview with urgent repayment work and summary context.

home
Primary action

Review overdue payments

Key content

Triage widgets + top lists

Dashboard follows the chosen Ops Console shell and should feel like the daily starting point for repayment work.

3. Borrowers List

Search-first table view for lookup, intake checks, and quick entry into borrower history.

list
Primary action

Create borrower

Key content

Search + filtered table

Phone-number search should be visually and functionally dominant, since borrower lookup is a core starting action.

4. Borrower Detail

Single borrower anchor page with summary, linked history, and next actions.

detail
Primary action

Create application

Key content

Identity + history + related records

This page should answer: who is this borrower, what has happened before, and can I safely create a new application?

5. Create / Edit Borrower

Simple, explicit data-entry form with duplicate-awareness and clear save path.

form
Primary action

Save borrower

Key content

Identity fields + guidance

Form should be straightforward, with special emphasis on phone uniqueness and actionable validation messages.

6. Applications List

Workflow-oriented application queue with clear state filtering and review readiness.

list
Primary action

Review next application

Key content

Status-driven queue

Unlike payments, this queue leans on review status and required next step, but still follows the same list grammar as other entities.

7. Application Review Detail

Stage-aware detail page with review steps, borrower history, and decision actions.

detail
Primary action

Key content

Review steps + borrower context

The page should preserve orientation: current application state, completed steps, remaining steps, and visible borrower history before a final decision.

8. Loans List

Lifecycle-driven overview of created, active, overdue, and closed loans.

list
Primary action

Open loan record

Key content

Status-heavy table

Loan list emphasizes lifecycle state and relationship to repayment status, not just raw loan metadata.

9. Loan Detail

Core record page for state, repayment schedule, documents, and linked records.

detail
Primary action

Next valid lifecycle action

Key content

Summary + schedule + linked context

This page must make loan state, payment schedule, and immutability rules visible without forcing the user to hunt through tabs.

10. Documentation & Disbursement

Focused progression page for pre-money readiness and guarded transition into disbursement.

action

Key content

Prerequisites + risk cues

Treat this as the last pre-money checkpoint: checklist-driven, explicit, and calm before the guarded confirmation step.

11. Payments List

High-frequency repayment workspace with overdue, upcoming, and recently completed filters.

Primary action

Open payment detail

Key content

Fast repayment triage
